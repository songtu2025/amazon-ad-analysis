# Amazon SP 广告健康监控与决策辅助系统开发文档

## 1. 项目定位

本系统面向 Amazon FBA 精品运营卖家，第一版只聚焦 `SP 商品广告`。

系统目标不是让 AI 自动执行广告动作，而是：

- 提高运营发现异常和判断问题的效率
- 监控 SP 广告健康状态
- 基于规则先判断异常和建议等级
- 使用 AI 解释规则结果、整理证据和风险
- 由人工决定采纳、修改后采纳、拒绝、加入观察或标记已处理
- 保留处理记录和复盘结果

一句话定义：

```text
SP 广告健康监控 + 规则判定 + AI 解释 + 人工决策 + 复盘记录
```

## 2. 第一版范围

### 2.1 已确认范围

- 广告类型：只做 `SP 商品广告`
- 判断周期：默认 `近 14 天`
- 数据来源：积加开放平台只读 API
- 数据库：第一版使用 `SQLite`
- 产品目标：由人工设置
- 建议颗粒度：
  - 产品层：承载经营目标
  - 关键词 / 搜索词层：承载具体异常和建议
- AI 角色：只做异常解释和决策辅助
- 人工处理动作：
  - 采纳建议
  - 修改后采纳
  - 拒绝并记录原因
  - 加入观察
  - 标记已人工处理

### 2.2 第一版不做

- 不自动修改广告预算、竞价、否词或暂停广告
- 不接广告写操作接口
- 不做 SB / SD 广告
- 不做多用户复杂权限
- 不做复杂预测模型
- 不做选品系统
- 不做全平台经营中台

## 3. 产品目标分类

每个产品第一版只设置一个当前主目标。

| 目标 | 含义 | AI 建议方向 |
| --- | --- | --- |
| 测试词款 | 找有效搜索词，允许低预算试错 | 低预算测试、观察、转精准 |
| 放量款 | 扩大有效流量 | 提预算、提竞价、保留高转化词 |
| 利润款 | 控 ACOS、提高净利 | 降竞价、否浪费词、缩预算 |
| 排名承接款 | 用广告推动核心词自然排名 | 保核心词、看转化和排名承接 |
| 清库存款 | 动销优先 | 保动销词、控制亏损风险 |
| 止损款 | 清理浪费 | 降竞价、否词、暂停观察、缩预算 |

产品目标由人工设置。AI 可以提示“目标与当前表现不匹配”，但第一版不自动修改产品目标。

## 4. 页面范围

第一版采用 5 页方案。

### 4.1 广告健康驾驶舱

用途：运营每天进入系统后的总览页。

核心内容：

- 同步状态
- 店铺 / 站点 / 产品目标 / 异常类型筛选
- 近 14 天健康概览
- 异常产品数量
- 待处理建议数量
- 高风险异常
- 浪费花费
- ACOS / CVR / 花费趋势
- 异常队列入口

### 4.2 产品目标与规则设置

用途：人工维护产品经营目标和判断门槛。

核心内容：

- 产品列表
- ASIN / MSKU / 产品名称
- 当前产品目标
- SP 花费、ACOS、CVR、订单、库存状态
- 目标与数据表现是否匹配
- 按产品设置规则门槛
- 保存设置 / 批量应用 / 放弃更改

### 4.3 异常与 AI 建议队列

用途：集中处理待判断事项。

核心内容：

- 异常列表
- 建议等级：
  - 可采纳
  - 小步测试
  - 观察
  - 禁止建议
- 异常类型
- 产品目标
- 命中对象：Campaign / Ad Group / Keyword / Search Term
- 关键指标
- 风险等级
- 进入详情页

### 4.4 建议详情与溯源

用途：人工确认前查看证据和风险。

核心内容：

- 建议标题
- 产品目标
- 命中关键词 / 搜索词
- Campaign / Ad Group
- 建议动作
- 规则判定结果
- AI 解释
- 数据快照
- 原始来源
- 风险提示
- 人工处理动作：
  - 采纳建议
  - 修改后采纳
  - 拒绝并记录原因
  - 加入观察

### 4.5 处理记录与复盘

用途：记录人工判断，并在后续复盘。

核心内容：

- 处理时间
- 处理人
- 原建议
- 人工处理结果
- 修改内容
- 拒绝原因
- 复盘状态
- 7 天复盘
- 14 天复盘
- 复盘指标变化

## 5. 技术栈

### 5.1 前端

```text
React
TypeScript
Vite
Ant Design
ECharts
```

选择理由：

- React + TypeScript 适合做复杂运营后台
- Vite 启动快，适合本地开发
- Ant Design 适合表格、筛选、表单、抽屉、弹窗等后台场景
- ECharts 适合趋势图、健康分布、异常图表

### 5.2 后端

```text
FastAPI
SQLite
SQLAlchemy
pandas
python-dotenv
httpx
APScheduler
```

选择理由：

- FastAPI 适合快速开发 API
- SQLite 适合第一版本地验证
- SQLAlchemy 方便后续切换 PostgreSQL
- pandas 适合广告数据清洗和聚合
- python-dotenv 用于读取本地密钥配置
- httpx 用于请求积加开放平台
- APScheduler 用于本地定时同步，第一版比 Celery 更简单

### 5.3 AI 层

第一版建议设计为独立服务函数，不直接嵌入规则判断。

```text
规则引擎先输出结构化结果
AI 只负责解释、风险说明、证据整理和建议文案
```

## 6. 系统架构

```text
积加开放平台只读 API
  -> 数据同步服务
  -> SQLite
  -> 规则引擎
  -> 异常事件
  -> AI 解释服务
  -> AI 建议
  -> 前端页面
  -> 人工处理记录
  -> 复盘记录
```

关键原则：

- 规则引擎和 AI 层分开
- 规则负责底线判断
- AI 负责解释和辅助决策
- 人工负责最终判断
- 所有建议必须可溯源

## 7. 积加开放平台 API 对接

### 7.1 第一版只读接口

第一版优先接以下接口。

| 数据 | 接口 |
| --- | --- |
| SP 关键词报表 | `/operation/ads/spKeywordsPage/query` |
| SP 商品报表 | `/operation/ads/adsSpProduct/query` |
| SP 广告活动 | `/operation/ads/adsSpCampaigns/query` |
| SP 广告位 | `/operation/ads/adsSpPlacement/page` |
| SP 用户搜索词-投放 | `/operation/ads/spSearchTargetingReport/page` |
| SP 用户搜索词-关键词 | `/operation/ads/spSearchKeywordsReport/page` |
| SP 商品投放 | `/operation/ads/adsSpProductTargeting/query` |
| 产品列表 | `/purchase/goods/product/page` |
| 产品详情 | `/purchase/goods/product/detail` |
| SKU 关联 MSKU | `/purchase/goods/amazonMsku/page` |
| 产品库存 | `/purchase/store/inventory/page` |
| FBA 库存 | `/purchase/store/fbaInventory/page` |
| 销售利润分析 | `/operation/sts/saleProfit/page` |

公开接口实际请求地址为：

```text
GERPGO_BASE_URL + /open + apiUrl
```

例如：

```text
https://open.gerpgo.com/api/open/operation/ads/spKeywordsPage/query
```

### 7.2 认证配置

本地 `.env` 示例：

```env
GERPGO_BASE_URL=https://open.gerpgo.com/api
GERPGO_APP_ID=your_app_id
GERPGO_APP_KEY=your_app_key
# 可选：如已手动获取 accessToken，可填写；留空时系统会用 AppID/AppKey 自动获取。
GERPGO_ACCESS_TOKEN=
GERPGO_MARKET_IDS=1,2,3
DATABASE_URL=sqlite:///./data/app.db
OPENAI_API_KEY=your_openai_api_key
```

注意：

- 不要把 `.env` 提交到代码仓库
- 积加接口可能要求 IP 白名单
- SP 广告接口需要 `marketId`
- 第一版先手动配置 marketId
- 官方 `获取accessToken` 接口为 `/api_token`，实际请求地址为 `GERPGO_BASE_URL + /open/api_token`
- `获取accessToken` 请求体需要 `appId` 和 `appKey`
- SP 广告业务接口请求头需要 `accessToken`

### 7.3 数据同步策略

第一版同步方式：

- 手动点击同步
- 可选本地定时同步
- 默认同步近 14 天数据
- 同步成功后写入 SQLite
- 同步状态展示在广告健康驾驶舱

分页规则：

- 优先使用接口返回的 `extObj` / `nextId`
- 每页数量按接口限制设置
- 第一阶段 smoke 默认使用 `count=10`、`max_pages=3`
- 分页请求之间保留约 `1.2s` 间隔，避免触发接口限流
- 同步日志记录每次请求结果

### 7.5 同步错误处理

| 场景 | 现象 | 处理方式 |
| --- | --- | --- |
| token 未配置 | smoke 启动时报配置缺失 | 检查 `.env` 中 `GERPGO_APP_ID`、`GERPGO_APP_KEY` 或 `GERPGO_ACCESS_TOKEN` |
| token 获取失败 | `/api/open/api_token` 返回非 200 或无 `accessToken` | 检查 AppID/AppKey、账号认证、IP 白名单 |
| 业务接口限流 | HTTP `509`，响应 `code=90008` | 降低 `count`、限制 `max_pages`、分页间隔不低于约 `1.2s` |
| API 无权限 | 响应中出现 `api没有权限` 或类似提示 | 到积加开放平台确认接口权限 |
| marketId 错误 | 返回空数据或参数错误 | 确认店铺 / 站点 `marketId` |
| 字段变化 | 落库字段为空或异常 | 保留 `raw_json`，先用原始响应校准字段映射 |

第一阶段不要一开始做全量同步。先用小页数验证链路，再逐步扩大同步范围。

## 9. 规则引擎

### 9.1 规则输出结构

规则引擎输出结构化结果，供 AI 解释和前端展示。

```json
{
  "anomaly_type": "acos_worse",
  "severity": "high",
  "suggestion_level": "adoptable",
  "object_type": "keyword",
  "object_id": "keyword_id",
  "matched_rules": [
    {
      "rule": "acos_above_target",
      "result": "failed",
      "message": "ACOS 高于产品目标门槛"
    }
  ],
  "blocked_rules": [],
  "evidence": {
    "period": "last_14_days",
    "clicks": 84,
    "cost": 126.4,
    "orders": 2,
    "sales": 79.8,
    "acos": 1.58,
    "cvr": 0.024
  }
}
```

### 9.2 建议等级

| 等级 | 含义 |
| --- | --- |
| 可采纳 | 数据量足够，规则方向明确，风险可控 |
| 小步测试 | 有信号，但样本或指标存在不确定 |
| 观察 | 样本不足，不建议马上动作 |
| 禁止建议 | 触发硬规则，AI 不应给采纳建议 |

### 9.3 硬规则示例

第一版规则可以先做成可配置。

```text
样本不足时，不能给可采纳建议
库存不足时，不能建议放量
利润款 ACOS 超目标时，不能建议加预算
清库存款可以容忍低利润，但不能忽略亏损风险
放量款必须同时满足 CVR、利润空间、库存承接，才允许提预算或提竞价
点击多无订单时，优先观察、降竞价、转精准或否词，不直接扩量
```

### 9.4 第一版异常规则

第一版规则先做成可配置，默认值只用于跑通系统，不代表最终经营门槛。

#### 9.4.1 ACOS 异常

用途：识别利润款、止损款中成本明显失控的关键词。

输入指标：

```text
cost
ads_sales
acos
clicks
ads_orders
product_goal
target_acos
```

触发条件：

```text
product_goal in ["profit", "stop_loss"]
clicks >= min_clicks
cost >= min_spend
acos > target_acos
```

建议等级：

```text
点击和花费样本充足 -> 可采纳
点击或花费接近门槛 -> 小步测试
样本不足 -> 观察
```

人工处理方向：

```text
降低竞价
转精准
否定明显无关搜索词
加入观察
```

#### 9.4.2 点击多无订单

用途：识别消耗点击但没有转化的关键词或搜索词。

输入指标：

```text
clicks
cost
ads_orders
cvr
keyword_text
search_term
product_goal
```

触发条件：

```text
clicks >= min_clicks
cost >= min_spend
ads_orders = 0
```

建议等级：

```text
点击和花费明显超过门槛 -> 可采纳
刚达到门槛 -> 小步测试
未达到门槛 -> 观察
```

人工处理方向：

```text
无关词：拒绝继续投放或否定
相关词：降竞价或加入观察
重要核心词：不直接否定，先看 Listing 和转化承接
```

#### 9.4.3 花费异常

用途：识别近 14 天广告花费明显偏高但结果不匹配的对象。

输入指标：

```text
cost
ads_sales
ads_orders
acos
roas
product_goal
```

触发条件：

```text
cost >= min_spend
且 ads_orders 低于产品规则门槛
或 acos 高于目标门槛
```

建议等级：

```text
高花费 + 低订单 + 高 ACOS -> 可采纳
高花费 + 有订单但利润承压 -> 小步测试
高花费但转化正常 -> 观察
```

人工处理方向：

```text
检查预算分配
降低低效关键词竞价
保留高转化词
缩减浪费词
```

#### 9.4.4 CVR 下滑

用途：识别转化率明显变差的关键词或广告组。

输入指标：

```text
cvr
clicks
ads_orders
cost
product_goal
```

触发条件：

```text
clicks >= min_clicks
cvr < target_cvr
```

建议等级：

```text
CVR 低且 ACOS 高 -> 可采纳或小步测试
CVR 低但样本少 -> 观察
```

人工处理方向：

```text
相关词：小幅降竞价并观察
不相关词：拒绝继续投放或否定
核心词：先检查 Listing、价格、库存、Review、主图
```

#### 9.4.5 曝光异常

用途：识别核心关键词曝光突然不足或投放状态异常。

输入指标：

```text
impressions
clicks
bid
serving_status
product_goal
```

触发条件：

```text
product_goal in ["scale", "rank_carryover", "test_keywords"]
且 impressions 明显低于规则门槛
```

建议等级：

```text
投放状态异常 -> 可采纳
竞价可能不足 -> 小步测试
样本不足或目标不需要扩量 -> 观察
```

人工处理方向：

```text
检查投放状态
检查预算是否不足
对核心词小幅提高竞价
非核心词只观察
```

### 9.5 规则实现顺序

第一版实现顺序：

```text
1. 点击多无订单
2. ACOS 异常
3. 花费异常
4. CVR 下滑
5. 曝光异常
```

原因：

- 前两类最容易从 SP 关键词报表直接判断
- 花费和 CVR 需要结合产品目标和规则门槛
- 曝光异常后续最好结合广告活动、预算和广告位数据再完善

## 13. 开发顺序

### 13.1 本地开发运行说明

从零启动第一阶段后端同步验证：

```powershell
Copy-Item .env.example .env
```

然后编辑 `.env`，至少填写：

```env
GERPGO_APP_ID=真实 AppID
GERPGO_APP_KEY=真实 AppKey
GERPGO_MARKET_IDS=真实 marketId
```

如已手动获取 `accessToken`，可以填写：

```env
GERPGO_ACCESS_TOKEN=真实 accessToken
```

运行 SP 关键词同步 smoke：

```powershell
python -B scripts\smoke_sync_sp_keywords.py
```

检查 SQLite 记录：

```powershell
python -B -c "import sqlite3; con=sqlite3.connect('data/app.db'); print(con.execute('select count(*) from sp_keyword_metrics').fetchone()[0]); print(con.execute('select status, rows_synced, period_start, period_end from sync_runs order by id desc limit 3').fetchall()); con.close()"
```

启动后端服务：

```powershell
cd backend
python -m uvicorn app.main:app --reload
```

健康检查：

```text
GET http://127.0.0.1:8000/health
```

手动触发同步：

```text
POST http://127.0.0.1:8000/api/sync/sp-keywords
```

常见问题：

| 问题 | 处理 |
| --- | --- |
| `GERPGO_APP_ID still contains a placeholder value` | `.env` 仍是示例值，换成真实值 |
| `接口调用次数已超过限制次数` | 等待后重试，降低页数，保留分页间隔 |
| SQLite 在沙箱内 `disk I/O error` | 在正常本地 PowerShell 运行；这是沙箱文件锁限制，不是代码结构问题 |
| 同步成功但数据为 0 | 检查 `marketId`、时间范围和接口权限 |
| 返回 API 无权限 | 到积加开放平台确认套餐或接口授权 |

### 阶段 1：项目骨架

```text
创建 FastAPI 后端

```

阶段 1 的最小验收目标：

```text
python scripts/smoke_sync_sp_keywords.py
```

脚本应能：

- 读取 `.env`

- 初始化 SQLite 表
- 拉取指定 marketId 的近 14 天 SP 关键词报表
- 第一阶段 smoke 默认限制为 `count=10`、`max_pages=3`，避免一开始全量分页触发接口限制
- 积加接口默认限流约为每秒 1 次；分页请求之间保留约 `1.2s` 间隔
- 将原始响应保存到 `data/raw/`
- 将指标写入 `data/app.db`

## 15. 风险与待确认事项

### 15.1 风险

- 积加 API 权限可能不包含全部广告报表
- IP 白名单可能导致本地请求失败
- marketId 需要先确认
- 产品、MSKU、ASIN、Campaign 之间映射可能不完整
- 利润和库存接口如果缺权限，放量判断会不完整
- 规则门槛需要用真实产品逐步校准

### 15.2 待确认

- 第一批接入哪些店铺和站点
- 第一批接入哪些产品
- 每个产品的当前主目标
- 是否有产品图片字段可用
- 利润数据是否能从积加 API 拉到
- 库存数据是否能稳定关联到产品
